## summary
The Ableton Move is a well-documented hacking target for a live-looping instrument. Control surface: 32 velocity-sensitive silicone pads with polyphonic aftertouch (4 rows x 8), 9 touch-sensitive endless encoders, a touch-sensitive clickable jog wheel, 16 multifunctional step buttons plus dedicated function buttons, and a 128x64 OLED — no touch strip. Audio: built-in mono "lo-fi but usable" sampling mic, 3.5mm stereo line-in and 3.5mm stereo out, internal speaker, class-compliant 2-in/2-out USB-C audio interface mode; the USB-A host port is MIDI-only (no USB audio expansion, unlike Push 3); the engine runs 44.1 kHz/16-bit. Compute: a Raspberry Pi CM4-class Broadcom BCM2711 (quad Cortex-A72 @ 1.5 GHz), 2 GB RAM, 64 GB internal SD, running a custom Linux with officially tolerated developer SSH access — the basis of mature community frameworks (extending-move, Move-Extended, Schwung/"Move Everything"). The audio path community code hooks is not ALSA/JACK but an SPI mailbox the `MoveOriginal` binary (SCHED_FIFO 70) drives at 44.1 kHz stereo int16 in 128-frame blocks (~2.9 ms/block); Schwung intercepts it via an LD_PRELOAD shim and already ships a bar-locked "Quantized Resampler" looper and shadow FX chains, though stock firmware "can live-sample but not really live-loop" (CDM) — firmware 2.0 (Feb 2026) added audio tracks, warping, and one-way Link Audio out. No measured round-trip latency figures were found anywhere; the 128-frame block size suggests low per-block latency but end-to-end monitoring latency is unverified. Battery is officially "up to 4 hours" and drains faster under heavy DSP, so a live set needs USB-C PD power (7.5 W+) at the venue.

# acf0c12f7e8810e3e

## summary
Schwung is a real, active community project: an unofficial open framework (formerly "Move Everything", itself built on bobbydigitales' "move-anything" hack) that runs custom synths, FX, MIDI tools, and full-screen "overtake" apps on the Ableton Move alongside — not replacing — stock firmware. It is not a firmware replacement: an LD_PRELOAD shim (schwung-shim.so) is injected into the stock MoveOriginal process and intercepts the SPI mailbox (mmap/ioctl) to mix shadow audio, filter/inject MIDI, composite the 128x64 display, and drive LEDs, while a forked QuickJS "shadow_ui" process hosts modules. Modules are written in modern JavaScript (ES2023, QuickJS) for UI plus optional native C/C++ .so DSP plugins (44.1 kHz, 128-frame stereo interleaved int16 blocks, ~2.9 ms), with documented APIs for audio render/process callbacks, internal/external MIDI send+receive, pad notes 68-99, setLED/setButtonLED, jog-wheel relative encoder decoding, and a procedural or object display API. Code execution comes via Move's own developer SSH interface (move.local/development/ssh); installation is a desktop installer or curl script, and modules are managed through the Schwung Manager web UI at move.local:7700 or dropped as tarballs into /data/UserData/schwung/modules/. A ~96-module catalog already includes several loopers and samplers (Magneto tape looper, TwinSampler, MrSample, MrDrums, Sample Slicer, Verglas/Clouds, Granny granular, DJ Deck) plus built-in Quantized Sampler and Skipback recording. Hardware is a quad-core Cortex-A72 @ 1.5 GHz with 2 GB RAM (reportedly a Raspberry Pi CM4), and realtime code on the SPI callback must be non-allocating and non-blocking.

# aa71030c7253c26e1

## summary
The Ableton Move hacking scene is mature and centers on three community frameworks plus one official one: (a) move-anything (bobbydigitales) — the original framework that discovered the technique of LD_PRELOAD-hooking the stock Move firmware's mmap/ioctl calls to hijack its 4KB SPI "mailbox" (shared memory carrying audio, MIDI, and display data), with a C core + QuickJS (ES2023) scripting layer; (b) Schwung (charlesvestal, formerly "Move Everything", built on move-anything, MIT) — a full "shadow mode" platform with ~90 installable modules including several loopers (Magneto tape looper, Ambiotica, War Bells, Clouds/Verglas), samplers (TwinSampler, quantized sampler, AutoSample, 30s–5min skipback buffer), and sequencers (Movy, dAVEBOx, Euclidean tools); (c) extending-move — a Python web server at move.local:909 that only manipulates files over SSH; and (d) RNBO Move Takeover — Cycling '74/Ableton's official experimental-alpha path for running Max/RNBO patches on the device. Architecturally, all deep hacks run ALONGSIDE the unmodified stock firmware process (never replacing it): the shim composites shadow audio into the hardware mailbox at 44.1kHz/128-frame blocks (~3ms), reads internal control-surface MIDI (cable 0) from the same buffer, and sidecar processes (shadow_ui JS host, chain_host dlopen'ing native C DSP .so modules, a Link Audio subscriber) communicate over named POSIX shared memory. Latency is handled pragmatically: Schwung synths render a few ms ahead of stock Move audio, with an optional ~16ms "Latency Comp" delay to align them when Link Audio per-track routing is on, and record-start is bar-quantized to MIDI clock/project tempo rather than latency-compensated. Licensing is split: move-anything is CC BY-NC-SA 4.0 with explicit anti-AI clauses, while Schwung is MIT and openly AI-agent-written, with GPL-3.0 module ports distributed free via a module-catalog store and GitHub releases. Bricking risk is considered low because Ableton officially ships SSH developer access (move.local/development/ssh) and a DFU restore mode; the main pitfalls are firmware updates disabling the hack until reinstalled, the need to back up sets first, instability/crashes, and unauthenticated web services on the LAN.

## summary
Loopy Pro's model: tap an empty clip to record (trigger-on-release by default), tap to close and play, two-finger tap to overdub; count-in/out quantization is per-clip (None=free/immediate, Master=next master-cycle boundary, Loop=multiple of the clip's own length for overdubs, Custom=user interval), and by default no tempo exists until the first recorded loop defines the tempo and master cycle length (a fixed tempo can instead be pre-set in the tempo panel); different-length loops stay in sync by being constrained to multiples/divisions of the master cycle; colors are DAW-style channels carrying insert FX and sends, with clip settings inheritable at project/color/clip levels; visual status is colored background=playing, black=stopped, dimmed=muted, orange-red=recording, wiper animation=queued transition; loops are drawn circular and one-shots square (play once). The OP-Z has exactly 12 punch-in effects mapped to one octave of keys (F to E): duck, filter sweep, loop 1, stereo, loop 2, pitch, follow/echo, ramp up/fill 1, short, ramp down/fill 2, long, random — momentary while held (shift+key; low octave = current track, high octave = whole drum or synth group), latched only by recording them (hold rec or record lock) onto the dedicated performance track where they become fully editable steps. Chord/harmony on OP-Z: the chord track is 4-note polyphonic per step, the master track analyzes bass/lead/arp/chord tracks to infer key and lets you sequence transpositions/chord progressions that selected tracks follow, with the per-step "tonality" step component controlling how each track responds. Live-looping performers follow a consistent layering order — beatboxed/drum loop first, then bass (Dub FX derives it from his voice via a SYB-5 bass synth pedal), then chords/harmonies, lead vocals last — with vocal-mic FX either printed into the loop pre-looper (Dub FX's GT-10) or per-function channels with sidechain ducking and linked macro FX (Beardyman's Beardytron); Marc Rebillet does the same on an RC-505's five stereo tracks plus MIDI keyboard, muting/unmuting tracks for arrangement.

## facts
- [confirmed] Loopy Pro loop lifecycle: tapping an empty clip starts recording and tapping again stops recording and starts playback; recording is triggered on finger RELEASE by default for timing accuracy (a 'Record on Press' gesture option exists); a two-finger tap toggles overdub on a clip; tap on a playing clip toggles its play state.
   src: https://loopypro.com/manual/ and https://wiki.loopypro.com/Introduction_to_Loopy_Pro
- [confirmed] Loopy Pro count-in quantization options per clip: None = recording starts immediately (subject to threshold recording); Master = starts at the beginning of the next master clock cycle; Loop = uses Master count-in when the clip is empty, but waits for the next multiple of the loop's own length to begin overdubbing when the clip has audio; Custom = user-defined interval (e.g. 1 bar).
   src: https://wiki.loopypro.com/Working_with_Record_Count-In/Out_and_Quantization
- [confirmed] Loopy Pro count-out quantization mirrors count-in: None = stop immediately; Master = stop when recording reaches the master cycle length or a multiple of it; Loop = overdub stops at the end of the loop's current cycle; Custom = stop at the set length or a multiple thereof. Length quantization additionally constrains loop lengths to multiples or subdivisions of a bar.
   src: https://wiki.loopypro.com/Working_with_Record_Count-In/Out_and_Quantization and https://wiki.loopypro.com/Set_Loop_Lengths
- [confirmed] Loopy Pro tempo/clock: no tempo is set at launch; by default the first recorded clip sets the tempo and the master cycle length ('in most live looping environments, the first loop sets the tempo'); alternatively a fixed tempo can be set beforehand in the tempo panel (needed for MIDI sequencers/tempo-synced FX); the master cycle length remains dynamically adjustable via the clock controls ('Bars' indicator).
   src: https://loopypro.com/manual/ and https://wiki.loopypro.com/Introduction_to_Loopy_Pro
- [likely] Loopy Pro sync of different-length loops: clips coordinate against the shared master cycle — recordings and overdubs align to master-cycle multiples (or the clip's own length multiples in Loop mode), so differing lengths stay phase-locked; individual clips can also be given a pre-determined fixed length in clip settings.
   src: https://wiki.loopypro.com/Set_Loop_Lengths
- [confirmed] Loopy Pro visual status conventions: playing clips use their color as background; stopped clips have a black background; muted playing clips use a dimmer version of the clip color; recording clips have an orange-red background; queued/scheduled transitions are shown with a 'colour wiper' animation over the black background.
   src: https://loopypro.com/manual/
- [confirmed] Loopy Pro FX model: colors aggregate clip output and 'are like tracks in a traditional DAW' — each color has a mixer channel strip (volume, balance, mute, solo) and accepts insert effects and sends; clip play/record settings can be defined at three inheritance levels: project-wide, per-color, and per-clip.
   src: https://loopypro.com/manual/
